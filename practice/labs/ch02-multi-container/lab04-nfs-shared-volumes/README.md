# 🧪 LAB 04: NFS Shared Volumes
*Focus: Persistent Multi-[Pod](../../../../GLOSSARY.md#pod) Storage (Network File System)*

## 🎯 Lab Goal
In this lab, you will learn how to use an `nfs` volume to persist and share data between multiple Kubernetes Pods. Unlike `emptyDir`, NFS volumes preserve their data even when Pods are deleted.

## 🛍️ Mall Analogy
In the **Central Mall**, an `nfs` volume is the **Central External Warehouse**.
- **[emptyDir](../../../../GLOSSARY.md#emptydir) (Lab 03)** → A temporary box in the back of a single shop. If the shop closes, the box is thrown away.
- **NFS** → The massive, permanent warehouse building across the street. Even if your shop in the mall is destroyed and rebuilt, the warehouse and its goods remain intact. Multiple shops can send delivery trucks there at the same time.

---

## 📋 Part 1: Setting up the Warehouse (NFS Server)

We need an actual NFS server running before our Pods can mount it. We will install it on one of our cluster nodes (e.g., `node01` or `minikube-m02`).

### 1. SSH into the Target [Node](../../../../GLOSSARY.md#node)
```bash
ssh node01
```

### 2. Install the NFS Server
Run the provided setup script. Select defaults if prompted.
```bash
./nfs-server-install.sh
```
*(When finished, type `exit` to return to your main control plane terminal).*

---

## 📋 Part 2: Connecting the Shop to the Warehouse

### 1. Inspect the Manifest
Review the `pod-nfs-vol.yaml` file:
```bash
cat pod-nfs-vol.yaml
```
Notice the `volumes` section pointing to the NFS server's IP address (`172.30.2.2`). Make sure this IP matches the internal IP of the [node](../../../../GLOSSARY.md#node) where you installed the NFS server.

### 2. Deploy the [Pod](../../../../GLOSSARY.md#pod)
Create the [pod](../../../../GLOSSARY.md#pod) that will write to the external volume:
```bash
kubectl create -f pod-nfs-vol.yaml
```

### 3. Verify
Watch until the [pod](../../../../GLOSSARY.md#pod) status turns to `Running`:
```bash
kubectl get po
```

---

## 📋 Part 3: Inspecting the Warehouse Operations

You should see data written by the container into the external NFS server.

### 1. Mount the Share Locally (Control Plane)
Run the helper script to mount the remote NFS share directly to your main terminal at `/var/data`:
```bash
./nfs-helper.sh
```

### 2. Check the Files
```bash
cat /var/data/test
```
*(You will see the "some text" lines generated by the alpine [pod](../../../../GLOSSARY.md#pod)).*

---

## 📋 Part 4: The True Power (Multi-Writer)
*Challenge Step:* 
Create a second [pod](../../../../GLOSSARY.md#pod) identically configured to use the same NFS volume. Notice how both Pods can read/write to the volume simultaneously without interfering with each other, proving that the storage exists completely independent of the [Pod](../../../../GLOSSARY.md#pod) lifecycles.

---

## 🧠 Key Takeaways
- **Persistence**: Deleting the [Pod](../../../../GLOSSARY.md#pod) unmounts the volume but leaves the data absolutely untouched.
- **`ReadWriteMany`**: NFS is one of the few storage types that easily supports multiple Pods writing to the same data concurrently.
